Deploy a Rails app with Puma and NGINX – CentOS, Fedora

Difficulty: 2
Time: 30 minutes

Puma is a simple, fast, and highly concurrent HTTP 1.1 server for Ruby web applications. It is designed for running Rack apps. This article will help you to deploy a Rails application with default database PostgreSQL, using Puma and Nginx on CentOS and Fedora.

Change PostgreSQL Configuration

  1. Open the configuration file.
    sudo vim /var/lib/pgsql/9.4/data/pg_hba.conf
  2. Look for the line that says Database administrative at the end of the file and replace the existing one with the below configuration. 
    # Database administrative login by Unix domain socket
local   all             postgres                                peer
# TYPE  DATABASE        USER            ADDRESS                 METHOD
# "local" is for Unix domain socket connections only
local   all             all                                     peer
# IPv4 local connections:
host    all             all             127.0.0.1/32            md5
# IPv6 local connections:
host    all             all             ::1/128                 md5
# Allow replication connections from localhost, by a user with the
# replication privilege.
#local   replication     postgres                                peer
#host    replication     postgres        127.0.0.1/32            md5
#host    replication     postgres        ::1/128                 md5
  3. Disable SELinux.
    setsebool -P httpd_can_network_connect 1

Create a database user for production

  1. Create a postgres username for the database.
    sudo -u postgres createuser -s myapp
  2. Provide a password to the database user.
    sudo -u postgres psql
  3. Type the following command in the PostgreSQL prompt.
    \password myapp
  4. When prompted for the password, enter a desired password and confirm the same. Press \q to exit.
    \q # for exit
  5. Make sure that you are in Rails application’s root directory. If not, then change the directory to Rails application directory.
    cd /path/to/myapp
  6. Open the database configuration file i.e. database.yml.
    sudo vim config/database.yml
  7. Find pool: 5 in the default section of database.yml file. Just below the pool: 5 line, add database connection string to connect. 
    pool: 5
host: localhost
  8. Move to the production section where you will find the following lines: 
    production:
  <<: *default
  database: myapp_production
  username: myapp
  password: <%= ENV['MYAPP_DATABASE_PASSWORD'] %>

<<: *default this line will add configuration from default section
database section , type your database name here

    In the username section, check if the user is similar to what you have created earlier. If not, then change it as per your requirement.

Install the Pluggin for rbenv-vars

You should set a secret key and a database password by using environment variable before you deploy Rails application in production. Here we are describing a way to manage environment variables by using rbenv-vars plugin that loads password and a secret key at the runtime. Next, we are going to install and configure rbenv-vars for our application.

  1. Setup the rbenv-vars by changing directory to .rbenv/plugins, using the following command and clone the repository of rbenv-var from github.
    cd ~/.rbenv/plugins
    git clone https://github.com/sstephenson/rbenv-vars.git

    Ensure that you’ve configured / installed rbenv in your machine correctly.

  2. Set the environment variable by changing your directory to application directory and generate the secret key.
    cd ~/myapp
    rake secret
  3. Once you have generated the secret key, you will find a token key. Copy and replace that key. To do this, open .rbenv-vars file into your favorite editor and replace your key with your_generated_secret key.
    vim .rbenv-vars
    SECRET_KEY_BASE=your_generated_secret

    Environment variable that you set in this file will be read by your Rails application.

  4. Set the database password for connection.
    MYAPP_DATABASE_PASSWORD=production_database_password
  5. Set your database environment variable similar to the one that is defined in the database configuration file i.e database.yml.
  6. Save and Exit.
  7. Check the environment variables list that you have set in file by using the following command:
    rbenv vars

Setup production database

  1. Once your application is able to communicate with the PostgreSQL database, create the production database.
    RAILS_ENV=production rake db:create

    Note: If you could not connect with database, please export all environment manually that are in rbenv vars

  2. Generate the controller and database migration using following command:
    rails generate scaffold Task title:string note:text
  3. Migrate to the production database using following command:
    RAILS_ENV=production rake db:migrate
  4. Precompile the assets.
    RAILS_ENV=production rake assets:precompile
  5. Test the application environments, where you can bind your public IP too.
    RAILS_ENV=production rails server --binding=server_public_IP
  6. Now open the browser and type the below mentioned in the address bar http://server_public_IP:3000/tasks

Install Puma

  1. Go back to your Rails application server and terminate it by using ctrl+c.
  2. Open the Gemfile.
    vim Gemfile
  3. Add the following lines at the end of the file. Before you proceed, make sure that you are in the application’s root directory.
    gem 'puma'
    gem 'foreman'
  4. Save and exit from the editor.
  5. To install Puma and its dependencies use the bundler command.
    bundle install
  6. This will take some time for installation.

Configure Puma

  1. Check the CPU cores before you proceed to configure the Puma server. This will help us to set workers and other process assignment.
    grep -c processor /proc/cpuinfo
  2. Open the Puma configuration file.
    vim config/puma.rb
  3. Copy and paste below configuration in the Puma file. 
    #!/usr/bin/env puma

# app do |env|
#   puts env
#
#   body = 'Hello, World!'
#
#   [200, { 'Content-Type' => 'text/plain', 'Content-Length' => body.length.to_s }, [body]]
# end

environment 'production'
daemonize false

pidfile '/home/imuser/myapp/tmp/pids/puma.pid'
state_path '/home/imuser/myapp/tmp/pids/puma.state'

# stdout_redirect 'log/puma.log', 'log/puma_err.log'

# quiet
threads 0, 16
bind 'unix:///home/imuser/myapp/tmp/sockets/puma.sock'

# ssl_bind '127.0.0.1', '9292', { key: path_to_key, cert: path_to_cert }

# on_restart do
#   puts 'On restart...'
# end

# restart_command '/u/app/lolcat/bin/restart_puma'


# === Cluster mode ===

workers 2 #change workers as per your cpu core


# on_worker_boot do
#   puts 'On worker boot...'
# end

# === Puma control rack application ===

activate_control_app 'unix:///home/imuser/myapp/tmp/sockets/pumactl.sock'
  4. Once you have pasted the configuration lines, change the workers as per your CPU cores. Save your changes and exit.

Start, Stop and Restart commands for Puma

  1. Below are the commands to start/stop and restart the Puma services. Puma server must me running to execute rails application on browser using Nginx.
    cd ~/myapp
    mkdir tmp/sockets tmp/pids
  2. To start: 
    bundle exec puma -C config/puma.rb -d --pidfile=tmp/pids/puma.pid
-d is for Daemonizing...
  3. To Stop:
    kill -s SIGTERM `cat tmp/pids/puma.pid`
  4. To Restart:
    kill -s SIGUSR2`cat tmp/pids/puma.pid`

Install and Configure Nginx

  1. Install Nginx using yum.
    sudo yum install nginx
  2. Now open the default server block with a text editor. 
    cd /etc/nginx/conf.d/
sudo rails_app.conf
  3. Add following in the configuration file: 
    upstream myapp {
  server unix:///home/imuser/myapp/tmp/sockets/puma.sock fail_timeout=0;
}
server {
  listen 80 deferred;
  server_name coolexample.com www.coolexample.com;
  root /home/imuser/myapp/myapp/public;
  try_files $uri/index.html $uri @myapp;
  location @myapp {
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header Host $http_host;
    proxy_redirect off;
    proxy_pass http://myapp;
  }
  error_page 500 504 /500.html;
  error_page 502 /502.html;
  error_page 503 /503.html;
  client_max_body_size 4G;
  keepalive_timeout 10;
}
  4. Restart Nginx to put the changes into effect.
    sudo service nginx restart
  5. You can now access the production environment of your Rails application using the public IP address or FQDN of the server. Visit the application server in a web browser to access the tasks controller.
    http://server_public_IP/tasks
